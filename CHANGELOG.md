### 경험치 기록 — 레벨 입력 구조 변경 (레벨업 시 음수 계산 버그 수정)

기존에는 "캐릭터 레벨" 단일 필드였는데, 레벨업이 있으면 종료 경험치(%)가 시작 경험치보다
작아져(예: 95%→10%) 획득 경험치가 음수로 계산되는 문제가 있었습니다.

- `ExperienceRecord`/`ExperienceFormValues`: `level` → `startLevel`/`endLevel`로 분리
- 획득 경험치 계산식 변경: `(종료레벨-시작레벨)×100 + 종료% - 시작%`
- 검증 로직도 변경: 레벨업이 없으면 기존처럼 종료%≥시작% 검사, 레벨업이 있으면 종료%가
  시작%보다 낮아도 정상으로 허용
- 기존 저장 데이터 마이그레이션: 예전 `level` 필드만 있는 기록은 `startLevel=endLevel=level`로 변환
- 실제 브라우저에서 시작Lv50→종료Lv51(레벨업), 시작95%→종료10%로 저장해 정상 처리됨을 확인

## v1.0.1 — OCR 후처리 전면 재작성 + 실사용자 테스트 (2026-07-10)

### 진짜 발견한 버그: OCR 결과가 후처리 없이 그대로 저장됨

사용자가 준 실제 OCR 예시("+WOV 요정중활 1A", "창고0사망 시 EEO I REO I AHO", "종류/활",
"공격력(작은/큰)/2(+0)/2(+0)")를 `parseOcrTextToItemDraft`에 직접 실행해보니, **아이템명에
OCR 원문이 그대로 들어가고, 메타정보 줄이 옵션에 뒤섞이고, 스탯은 아예 인식이 안 되는
실제 버그를 확인했습니다.** `src/lib/itemParser.ts`를 전면 재작성했습니다.

- **Dictionary 자동보정** 추가 (`ocrCorrection.ts`): WOV→+0, VV→W, O→0, I→\|, l→1,
  EEO→드롭O, REO→교환O, AHO→삭제O
- **메타정보 줄 필터링**: "창고/사망/드롭/교환/삭제" 키워드 2개 이상 포함된 줄, "아이템 정보"
  같은 섹션 헤더는 스탯/옵션으로 저장하지 않고 제거
- **"라벨\n값" 2줄 패턴 지원**: 실제 인벤 상세페이지 구조("종류" 다음 줄에 "활")를 정확히 파싱
- **인챈트 분리**: "+7 사이하의 활" → 강화수치 "+7"과 이름 "사이하의 활"을 분리, 강화수치는 메모에 기록
- **THE K DB 자동매칭(유사도 90%+)**: `dbMatch` 필드로 반환, 매칭되면 아이템명/무기종류/공격력/
  원거리명중/원거리대미지/특수옵션/이미지/등급/필요레벨/안전인챈트/착용클래스 **전부 DB 값을
  우선 적용**하고 OCR은 강화수치(메모)만 반영 — DB에 없는 항목만 OCR 값을 그대로 씀
- **OCR 등록 미리보기 3단 패널** 신규 (`OcrPreviewPanel.tsx`): ① 자동 선택된 아이템 ② OCR 원본
  인식 결과 ③ 실제 적용 결과를 나란히 보여주고, 사용자가 확인 후 저장
- 옵션은 항상 줄 단위로 저장 (기존에도 그랬음, 재확인)
- 실제 tsx로 파서를 직접 실행해 결과를 검증(추측 아님): 메타줄 제거/인챈트분리/2줄패턴파싱/
  DB자동매칭 전부 정확히 동작 확인. 브라우저에서도 "새 아이템 등록 → 저장 → 목록 → 수정 →
  재저장" 전체 플로우를 실제 클릭/타이핑으로 검증 완료

### 기능별 적용 (5개 화면)

| 화면 | 내보내는/불러오는 데이터 |
|---|---|
| 경험치 (`Experience.tsx`) | 경험치 기록, 즐겨찾기 파티, 목표일 |
| 아데나 매입 (`AdenaPurchase.tsx`) | 환율/수량 설정, 매입 기록 |
| 아이템 비교 (`Items.tsx`) | 아이템 DB, 최근 검색어 |
| 제작 계산기 (`Estimate.tsx`) | 저장된 프리셋 전체 (기존에 있던 "현재 편집중인 견적" JSON 저장/가져오기는 그대로 유지, 별개 기능) |
| 설정 (`PreferencesForm.tsx`) | 환경설정 값 |

각 화면의 불러오기는 기존 `IMPORT_MERGE` 리듀서를 재사용하되, payload에 **해당 섹션 필드만**
채워 넘긴다 — `IMPORT_MERGE`가 원래 "payload에 없는 필드는 건드리지 않는" 구조라 별도의
새 리듀서 액션 없이도 안전하게 섹션 격리가 된다.

파일에는 `thekSection` 값이 들어있어, 예를 들어 "아이템 비교" 화면에서 "경험치" 파일을 잘못
선택하면 "이 파일은 '경험치' 데이터입니다. '아이템 비교' 파일을 선택해주세요"로 명확히 거부한다.

### 실제 실행 검증 (tsx로 실제 리듀서 실행)

5개 섹션 전부 개별적으로 다음을 실제 실행해서 확인했다:
1. 모든 섹션에 데이터가 있는 상태를 만든다
2. 특정 섹션 파일 하나만 `IMPORT_MERGE`로 병합한다
3. 그 섹션 데이터는 정확히 갱신되고, **나머지 4개 섹션은 값이 단 하나도 바뀌지 않았음**을 확인
4. 섹션이 다른 파일을 잘못 선택하면 명확한 오류 메시지와 함께 거부됨을 확인

```
[경험치]   기록 병합됨(1→2), 목표일 반영 | 아데나/아이템/프리셋/설정 전부 그대로
[아데나]   rate 갱신(500→1234)           | 경험치/아이템/설정 전부 그대로
[아이템]   아이템 병합(1→2)              | 경험치/아데나/설정 전부 그대로
[제작계산기] 프리셋 병합(1→2)             | 경험치 그대로
[설정]     theme 적용(light→dark)        | 경험치/아데나/아이템 전부 그대로
[검증] 섹션 불일치 파일("아데나" 파일을 "경험치"로 시도) → 정확히 거부됨
```

- 전역 TypeScript(strict) 재검증: 실제 오류 0건
- ⚠️ `npm install`/`npm run build`/`npm run dev` 자체는 여전히 이 환경의 npm 레지스트리

## 아데나 매입 "0원" 재현 - 실행 검증으로 원인 규명 (2026-07-10)

`tsx`로 실제 리듀서(`appDataReducer`)를 직접 실행해서 `PURCHASE_SETTINGS_PATCH` dispatch →
state merge → `liveTotal` 계산까지 전 과정을 추적했습니다.

```
rate=1050, currentAmount=2000 dispatch 후 → liveTotal = 2100000 (정상)
```

**데이터 레이어(리듀서/state/계산식)는 실행 결과로 100% 정상임을 확인했습니다.**
그래서 UI 쪽을 다시 봤더니, 유력한 원인을 찾았습니다:

**"실시간 총 매입금액"(환율×현재수량) 카드 바로 옆에 "총 사용금액"(매입 기록 합계) 카드가
있었는데, 이름이 비슷하고 나란히 붙어 있어 혼동하기 쉬운 배치였습니다.** 매입 기록을 하나도
등록하지 않은 상태라면 "총 사용금액"은 0원이 맞는데, 이걸 "실시간 총 매입금액"으로 착각하기
쉬웠을 것으로 보입니다.

- "총 사용금액" → **"누적 매입 등록 금액"**으로 이름 변경, "실시간 총 매입금액과는 다른 값이고
  등록된 기록이 없으면 0원이 정상"이라는 설명 문구 추가
- "실시간 총 매입금액" 카드를 2칸으로 넓히고 더 크게 강조, **계산에 쓰인 실제 값을 카드 안에
  그대로 표시**(`= 오늘 환율(1050) × 현재 수량(2000)`)해서 상태값이 제대로 들어왔는지
  화면에서 바로 확인 가능하게 함
- 계산 자체는 이전 라운드의 `Number()`/`useMemo` 방어 로직 유지

## 아데나 매입 실시간 총 매입금액 재수정 (2026-07-10)

사용자가 `npm run dev`로 직접 실행해 "현재 수량을 입력해도 0원"이라는 재현 결과를 주셔서
다시 조사했습니다. `PurchaseSettingsCards.tsx`/`AdenaPurchase.tsx`의 React state 흐름
(입력→dispatch→reducer 병합→context→재렌더)을 코드로 몇 번을 다시 봐도 논리적 결함을
찾지 못했지만, 혹시 모를 타입 이슈(예: 저장된 값이 문자열이거나 NaN인 경우)에 완전히
안전하도록 다시 작성했습니다.

- `liveTotal`(실시간 총 매입금액), `remainingAmount`(남은 수량) 계산을 `Number()` 명시
  강제변환 + `Number.isFinite` 체크 후 `useMemo`로 재작성 (`PurchaseSettingsCards.tsx`)
- `previewCashAmount`(매입 등록 미리보기)도 동일하게 방어적으로 강제변환 (`AdenaPurchase.tsx`)
- 이 환경에서는 실제 브라우저로 재현/확인이 불가능해 100% 해결을 장담할 수는 없습니다.
  만약 이번에도 0으로 보이면, "남은 수량"이나 "매입 등록"의 실시간 미리보기 값도
  같이 안 바뀌는지 확인해서 알려주시면 원인 범위를 좁히는 데 큰 도움이 됩니다.

## 최종 출고 정리 (2026-07-09)

### 수정한 기능

- **실제 빌드 실패 수정 (TS5101/TS5103)**: `tsconfig.json`의 `"ignoreDeprecations": "6.0"`이
  프로젝트가 쓰는 TypeScript 5.7.2에서는 유효하지 않은 값이라 `npm run build`가 아예
  실패했습니다. 원인을 조사해보니 `baseUrl` deprecation 자체가 **TypeScript 6.0부터
  적용되는 정책**이라 5.7.2에서는 애초에 필요 없는 옵션이었습니다. `ignoreDeprecations`뿐
  아니라 `baseUrl`도 함께 제거했습니다 (`paths`가 이미 `"./src/*"`처럼 상대경로 프리픽스를
  쓰고 있어서 TypeScript 4.1+ 정책상 `baseUrl` 없이도 그대로 동작합니다). 이렇게 하면
  앞으로 TypeScript 6.0으로 올려도 이 문제가 재발하지 않습니다. `typescript`도 버전
  범위(`^5.7.2`) 대신 `5.7.2`로 정확히 고정해 마이너 버전 차이로 인한 위험을 없앴습니다.
  실제 `tsc`로 재검증해 이 오류가 완전히 사라진 것을 확인했습니다.

- **아이템 비교 – 인벤 DB 연동**: 브라우저가 인벤 도메인에 직접 요청해 CORS(Failed to fetch)로
  막히던 문제를 프록시 경유 구조로 변경해 해결
- **아데나 매입 – 계산 방식**: 총 매입금액이 입금 체크와 무관하게 항상 실시간으로 계산되도록

- **TypeScript 빌드 오류 전수 수정** (strict + `noUncheckedIndexedAccess` 기준 0건까지):
